**玄铁，一门骨子里就是中文的现代通用编程语言。**

> 目前玄铁的开发团队由 1 人组成，即作者(问号盒)。作者负责设计、实现、测试和维护玄铁。AI辅助开发则负责生成代码、优化代码、修复错误等。

玄铁由中国广东一名 17 岁开发者独立设计并实现。从 2026 年 4 月 3 日的 v0.2 起步，目前正处于完善语法完善编译器底层的核心攻坚阶段。

玄铁不仅是简单的语法汉化，其核心哲学在于**“高熵语义下的逻辑降维”**：利用中文单字极高的信息密度作为语言原语（如 `设`、`若`、`听`、`求`），让代码逻辑骨架一眼到底。对于中文母语者，玄铁意味着零认知负荷的逻辑构建。

## 核心特性

- **极致精炼的语法**：关键字全面单字化。`设` (声明)、`函` (函数)、`返` (返回)、`听` (服务监听)、`求` (网络请求)、`道` (并发通道)。
- **现代类型系统**：支持可选类型注解、联合类型 (`字|整`)、空安全 (`字?`)、泛型与泛型约束。
- **面向对象进阶**：支持单继承 (`承`)、方法重写 (`覆`)、三级权限控制 (`公/护/私`) 及魔术方法重载。
- **Result 错误处理范式**：通过 `成功()`/`失败()` 包装返回值，支持 `.接着()`/`.否则()` 链式调用，彻底杜绝 `if err != nil` 金字塔。
<!-- - **原生高并发模型**：基于 Go 协程封装（种子编译器），通过 `异步`、`等待`、`并行` 实现自然语义下的高并发。 -->
- **系统级原语**：`听`、`求`、`连`、`执` 均为语言级关键字，而非外部库，实现网络与系统操作的“降维打击”。

## 来上手

### Hello World 与 字符串插值

```xuantie
设 名字 = "玄铁"
示("你好，#{名字}！1+1=#{1+1}")
```

### 函数式错误处理

```xuantie
函 获取数据(id) {
    若 id == 0 { 返 失败("无效 ID") }
    返 成功({"名": "玄铁", "值": 100})
}

获取数据(1).接着(函(对象) {
    示("获取到: " & 对象["名"])
}).否则(函(错) {
    示("错误: " & 错)
})
```

## 技术架构：自举与 LLVM

玄铁的技术演进经历了从**解释器**到 **Go 转译器**，再到如今 **LLVM 自举**的飞跃。

### 自举架构 (Bootstrapping)

玄铁的目标是实现“玄铁编玄铁”。目前架构由两部分组成：

1. **种子编译器 (Go)**：负责将玄铁源码编译为二进制，是自举的起点。
2. **自举编译器 (XuanTie)**：位于 `xuantie_compiler/` 目录，完全由玄铁编写，包含词法分析、语法分析与 LLVM IR 生成后端。

### 底层硬核原理

- **原子引用计数 (Atomic ARC)**：基于 C11 `stdatomic.h` 的内存管理，无 GC 停顿。
- **标记指针 (Tagged Pointers)**：所有变量统一为 `i64`，最低位区分指针与内联整数，内联整数实现零内存分配。
- **O(1) 内存池**：针对 AST 节点等小对象引入定长空闲链表内存池，极大降低 `malloc/free` 开销。
- **LLVM 深度优化**：通过 `mem2reg` 优化将栈操作转化为寄存器操作，算术运算在位级别完成，避免装箱拆箱。

## 为什么选择玄铁？


| 特性         | 玄铁 (XuanTie)           | 易语言               | Go / Python     |
| :------------- | :------------------------- | :--------------------- | :---------------- |
| **开放性**   | 开源且自举，任何人可贡献 | 封闭系统，硬编码 C++ | 工业标准        |
| **平台**     | 天生跨平台 (LLVM)        | 绑定 Windows         | 跨平台          |
| **语义**     | 高熵单字原语，逻辑直观   | 汉化外壳，底层非中文 | 英文缩写        |
| **并发**     | 语言级`异步/等待/道`     | 依赖 API 库          | 协程 / 线程     |
| **错误处理** | Result 管道化            | 传统异常             | `if err != nil` |

## 大致计划

1. **v1.0.0 正式自举**
   用玄铁写的编译器能编译玄铁代码本身。这是语言图灵完备性的终极证明。
2. **v1.1.0 渲染引擎**
   实现基于 `raylib` 或类似底层封装的玄铁原生渲染引擎。
3. **v2.0.0：玄铁铸造厂 (IDE)**
   用玄铁编写玄铁自己的 IDE。届时，玄铁的工具链（编译器、渲染引擎、IDE）将全部由玄铁自身编写。

> 在自举之前,玄铁的语法可能随时进行变动，不推荐在生产环境中使用。并且很可能会一直保持0.x版本(甚至0.99。虽然概率很低，但不是不可能。)。作者（问号盒）需保证自举编译器能完全稳定可用且语法真正定型后才会发布1.0版本。

## 其他

可能有人会觉得几个问题或疑问：

1. 单代码文件臃肿（xt_runtime、编译.xt 等）
   * 本作者也尝试过拆分物理文件，但是经过整整5个小时通宵处理再烧了20美刀的Token给Vibe处理也未能成功。编译器内部的设计逻辑太复杂了，牵扯到很多潜在Bug和问题，越弄会越乱越耗时。不过好在处理的过程中发现了不少潜在问题，我都一并修复了，增强了底层稳定性。
2. 部分代码编写像AI
   * 这个直觉是对的，有不少代码确实使用Trae辅助完成。不过请放心，所有Trae编写的代码都经过作者审查和处理后再丢给Claude-Opus4.7严格检查后发布的。
3. 作者更新时很快时很慢
   * 这个随缘更新，虽然说作者的初衷和本意以及目标的确是打造一个“真正可商用、可生产的通用中文编程语言”但在3.0之前本质上依然属于“玩具”范畴，并且本项目完全MIT开源，大家都是可以拿去玩和加入成为贡献者的。作者只会有空和有时间再打磨更新，但并不会停止维护的，请放心。
