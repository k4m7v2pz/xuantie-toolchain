# 自举编译器实现笔记

> 本文档是玄铁自举编译器（`xuantie_compiler/` 目录下由玄铁自身编写的编译器）的实现笔记，面向未来的维护者和协作者。
> 自举编译器由 Gemini 辅助开发完成主体架构，Claude 辅助种子编译器优化。本笔记基于 2026-05-04 实际代码审计编写，**v0.17.5 已修复 7.1\~7.3、7.7、7.10**。

***

## 1. 编译器总览与架构

### 1.1 编译流程

```
玄铁源码 (.xt) → 词法.xt → 语法分析.xt → 抽象语法树.xt → 编译.xt（四趟）→ LLVM IR → clang/gcc → 原生二进制
```

自举编译器是一个用**玄铁自身编写**的四趟编译器，核心文件：

| 文件                          | 行数      | 职责                               |
| --------------------------- | ------- | -------------------------------- |
| `词法.xt`                     | 462     | 词法分析，将 UTF-8 字符转为 Token 流        |
| `抽象语法树.xt`                  | 478     | 所有 AST 节点类型定义                    |
| `语法分析.xt`                   | 1310    | 递归下降 + Pratt Parsing 语法分析        |
| `编译.xt`                     | 3633    | **核心**：四趟编译——预扫描、语义分析、类型推导、IR 生成 |
| `玄铁.xt` / `xt.xt` / `入口.xt` | 110-222 | 驱动入口，CLI 参数解析、调用编译器、执行 clang/gcc |

### 1.2 与种子编译器的关键差异

| 维度    | 种子编译器 (Go)                                 | 自举编译器 (玄铁)                                       |
| ----- | ------------------------------------------ | ------------------------------------------------ |
| 编译趟数  | 单趟（代码生成即类型追踪）                              | 四趟（预扫描→语义→类型推导→IR）                               |
| 类型追踪  | 内联在 `compileExpression` 返回值                | 独立的 `推导程序类型` pass + `变量类型表`                      |
| 前向引用  | AST 节点天然支持                                 | 通过全局预扫描收集符号解决                                    |
| 模块系统  | `ImportExpression` 内联处理                    | 全局预扫描递归 + `程序缓存` + `语义分析` 拓扑排序                   |
| 运算符重载 | `InfixExpression` + `magicMethod` fallback | `编译中缀` 先试 `xt_dict_get` 动态分发，再 fallback `编译内建中缀` |
| 内存管理  | Go GC                                      | Arena 内存池（`xt_arena_new`/`xt_arena_use`）         |

### 1.3 编译器结构体关键字段

出自 [编译.xt L1-L40](../xuantie_compiler/编译.xt)：

```xuantie
型 编译器 {
    设 结果输出列表   // 当前函数/主域的 IR 行列表
    设 结果输出       // 最终拼接的完整 IR 字符串
    设 寄存器数       // 递增计数器，生成 %t1, %t2...
    设 标签数         // 递增计数器，生成 if.then.1, loop.cond.2...
    设 字符串映射     // 字符串常量去重 → 全局别名映射
    设 作用域栈       // [[addrReg...]]，每层作用域需要 ARC 释放的寄存器列表
    设 符号表         // 变量名 → alloca 寄存器地址或全局符号名
    设 变量类型表     // 变量名 → 类型（"raw_i64" / "i64" / "i1" / "型:类名"）
    设 函数表         // 函数名 → 真，标记已声明的函数
    设 类型表         // 类型名 → AST 节点，存储型定义
    设 函数输出       // 所有函数定义的 IR 文本
    设 外部函数输出列表 // 外部函数声明的 IR 行列表
    设 当前型名称     // 编译型定义方法时的上下文
    设 是全局         // 真 = 在主域，假 = 在函数/方法内
    设 已访引入       // 防止重复引入
    设 程序缓存       // 文件路径 → AST 程序节点，避免重复解析
    设 已访语义       // 防止语义分析死循环
    设 已访别名       // 追踪模块别名，防止被误当作函数
    设 主文件路径     // 用于相对路径解析
    设 断标签栈       // break 目标标签栈
    设 续标签栈       // continue 目标标签栈
}
```

## 2. 词法分析器（词法.xt）

位于 [词法.xt](../xuantie_compiler/词法.xt)，462 行。

### 2.1 Token 结构

```xuantie
型 标记 {
    设 标记类型 = ""   // Token 类型关键字，如 "标识符"、"数字"、"函数定义"
    设 标记字面量 = "" // 原始文本
    设 标记行 = 0      // 行号（1-based）
    设 标记列 = 0      // 列号（1-based）
}
```

### 2.2 词法分析器工作机制

**字符流模型**：将源码字符串转为字符列表 `字符列表`，通过 `读字符()` 单字符前进，`窥视字符()` 预读下一字符。

**关键字识别**（[词法.xt L409-L464](../xuantie_compiler/词法.xt#L409)）：所有中文关键字通过硬编码匹配表识别。例如：

```
"设" → "变量声明"、"函" → "函数定义"、"型" → "型定义"
"若" → "如果"、"遍历" → "遍历循环"、"匹配" → "匹配"
```

**字符串处理**（[词法.xt L230-L296](../xuantie_compiler/词法.xt#L230)）：

- 支持 `"..."` 和 `'''...'''` 两种引号风格
- 支持插值标记 `#{}`——词法器**不解析插值**，将整段字符串（含 `#{表达式}` 原文）作为一个 Token 返回，插值解析在语法分析阶段处理

**标识符规则**（[词法.xt L88-L114](../xuantie_compiler/词法.xt#L88)）：

- 排除所有 ASCII 符号和数字后，剩余字符（含所有多字节 UTF-8 字节）视为"字母"
- 支持以 `?` 结尾（如 `存在?`、`包含?`）

**注释**：`//` 到行尾，在 `读标记()` 中处理。

## 3. 抽象语法树（抽象语法树.xt）

位于 [抽象语法树.xt](../xuantie_compiler/抽象语法树.xt)，478 行。

### 3.1 基础节点

```xuantie
型 基础节点 {
    设 节点行       // 源码行号
    设 节点列       // 源码列号
    设 节点类别     // "函数定义"、"中缀"、"标识符" 等——编译器的核心分发依据
    设 推导类型     // 第三趟类型推导的结果（"整"、"字"、"逻"、"未知"）
}
```

**关键设计**：所有节点继承 `基础节点`，`节点类别` 是编译器 `匹配` 语句的**唯一分发键**。这是理解整个编译器控制流的核心——不是靠 Go 的类型 switch，而是靠字符串匹配。

### 3.2 关键节点类型速查

| 节点类型       | 节点类别      | 关键字段                        |
| ---------- | --------- | --------------------------- |
| `整数节点`     | `"整数"`    | `整数值`                       |
| `标识符节点`    | `"标识符"`   | `标识符名称`                     |
| `中缀表达式节点`  | `"中缀"`    | `中缀左`、`中缀运算符`、`中缀右`         |
| `前缀表达式节点`  | `"前缀"`    | `操作符`、`右节点`                 |
| `调用表达式节点`  | `"调用表达式"` | `函数表达式`、`调用参数`              |
| `成员访问节点`   | `"成员访问"`  | `左表达式`、`成员名称`               |
| `函数定义节点`   | `"函数定义"`  | `函数名`、`参数列表`、`返回类型`、`函数体块`  |
| `变量声明语句节点` | `"变量声明"`  | `声明名称`、`声明值`、`注解类型`         |
| `块语句节点`    | `"块"`     | `块语句列表`                     |
| `若语句节点`    | `"若语句"`   | `若条件`、`若结果块`、`抑分支列表`、`若否则块` |
| `遍历循环节点`   | `"遍历循环"`  | `迭代变量名`、`迭代目标`、`循环块`        |
| `引入表达式节点`  | `"引入"`    | `路径`、`别名`                   |

### 3.3 函数定义节点的参数表示

**重要**：在自举编译器的 AST 中，函数参数用 `标识符节点` 的列表表示，类型注解存在 `标识符节点.推导类型` 字段中（[语法分析.xt L502](../xuantie_compiler/语法分析.xt#L502)）：

```xuantie
p节点.推导类型 = 此.解析类型()  // 存的是原始类型名，如 "整"、"字"
```

这与种子编译器（Go）的 `Parameter.DataType` 不同——自举编译器**复用了 AST 节点上已有的** **`推导类型`** **字段**来存储参数类型注解，而不是新增一个专门的字段。

## 4. 语法分析器（语法分析.xt）

位于 [语法分析.xt](../xuantie_compiler/语法分析.xt)，1310 行。

### 4.1 架构

- **语句级**：`解析语句()` 通过 `匹配 类型` 分发（"变量声明"→解析变量声明、"函数定义"→解析函数定义……）
- **表达式级**：Pratt Parsing（算子优先级解析），`解析表达式(优先级)` + `解析中缀(左)`
- **前瞻**：`当前标记` + `窥视标记`（一个 token 的前瞻缓冲）

### 4.2 Pratt 优先级表

出自 [语法分析.xt L18-L27](../xuantie_compiler/语法分析.xt#L18)：

```
"或": 1,  "且": 2,  "等于": 3
"相等"/"不等"/"是": 4
"小于"/"大于"/"小于等于"/"大于等于"/"范围": 5
"加"/"减"/"连接"/"位或"/"异或": 6
"乘"/"除"/"取模"/"位与"/"左移"/"右移": 7
"左圆括号"/"左方括号": 8
"点"/"问号": 9
```

### 4.3 解析函数定义（[语法分析.xt L466-L528](../xuantie_compiler/语法分析.xt#L466)）

流程：

1. 解析函数名（允许 `造` 作为构造函数名）
2. 解析泛型参数 `<T>`
3. 解析参数列表 `(name: Type, ...)`
4. 解析返回类型注解 `: Type`
5. 解析函数体 `{ ... }`

**类型注解处理**：每个参数的 `推导类型` 字段被重用存储解析到的类型名（如 `"整"`），这是后续类型推导和代码生成的关键数据。

### 4.4 解析中缀（[语法分析.xt L651-L718](../xuantie_compiler/语法分析.xt#L651)）

| Token | 生成的 AST 节点          |
| ----- | ------------------- |
| `(`   | `调用表达式节点`           |
| `[`   | `索引表达式节点`           |
| `.`   | `成员访问节点`            |
| `=`   | `中缀表达式节点`（运算符="="）  |
| `是`   | `中缀表达式节点`（运算符="是"）  |
| `..`  | `范围表达式节点`           |
| `?`   | `问号后缀节点`            |
| 其他    | `中缀表达式节点`（运算符=原始文本） |

## 5. 编译器（编译.xt）—— 四趟编译详解

位于 [编译.xt](../xuantie_compiler/编译.xt)，3633 行，是自举编译器最大的文件。

### 5.1 编译入口：编译程序

```xuantie
函 编译程序(程序节点, 源文件路径) {
    // 第一趟：全局预扫描（收集符号）
    此.全局预扫描(程序节点, "", 源文件路径)
    // 第二趟：语义分析（类型成员解析 + 拓扑排序）
    此.语义分析(程序节点)
    // 第三趟：类型推导（Static Type Inference）
    此.推导程序类型(程序节点)
    // 第四趟：代码生成（IR Emit）
    // ...遍历程序语句列表，调用 编译语句()...
    // 最后拼接 IR 头部（类型声明、外部函数声明、字符串常量）+ 函数体 + main
}
```

### 5.2 第一趟：全局预扫描

函数位置：[编译.xt L239](../xuantie_compiler/编译.xt#L239)。

**职责**：在代码生成前收集所有顶层符号，解决前向引用问题。

**匹配的节点类别**：

- `"函数定义"` → 注册到 `函数表` 和 `符号表`
- `"型定义"` → 注册到 `类型表`，预扫描所有方法签名
- `"接口定义"` → 注册到 `类型表`
- `"外部函数"` → 调用 `编译外部函数定义` 生成声明
- `"表达式语句"` 且是 `"引入"` → 递归预扫描引入的模块（`预扫描引入`）
- `"变量声明"` → 全局变量注册到 `符号表`

**引入机制**：[预扫描引入](../xuantie_compiler/编译.xt#L300)：

1. 按主文件路径解析相对路径
2. 检查 `程序缓存`——若已解析过则复用 AST
3. 否则 `文件.读` + `造 词法分析器` + `造 语法分析器` + `解析程序()`
4. 将解析结果缓存到 `程序缓存`
5. 递归调用 `全局预扫描`，带上命名空间前缀

### 5.3 第二趟：语义分析

函数位置：[编译.xt L319](../xuantie_compiler/编译.xt#L319)。

**职责**：解析类型成员、处理继承链、插入 `此` 参数。

流程：

1. 递归处理引入模块的语义分析
2. 对型定义做拓扑排序（先解析父类，再解析子类）
3. `解析型成员` 中：
   - 继承父类的属性和方法列表（扁平化到 `最终属性列表` / `最终方法列表`）
   - 为每个方法的参数列表**插入** **`此`** **参数**（如果没有的话）
   - 处理方法重写：用 `原始函数名` 匹配，覆盖同名的父类方法

**关键细节**：语义分析修改了 AST 节点——给方法加了 `此` 参数后，函数定义的 `参数列表` 变化了。这影响后续类型推导和代码生成的参数遍历。

### 5.4 第三趟：类型推导

函数位置：[编译.xt L3405](../xuantie_compiler/编译.xt#L3405)。

**职责**：遍历 AST 树，为每个表达式节点打上 `推导类型` 标签。

**推导规则**：

- 字面量：`"整数"` → `"整"`、`"小数"` → `"小数"`、`"字符串"` → `"字"`
- 标识符：查 `变量类型表`
- 中缀表达式：加减乘除 → `"整"` 或 `"小数"`；比较/逻辑 → `"逻"`；连接 → `"字"`
- 调用表达式：查函数返回值类型（`变量类型表`）
- 成员调用：`.字节`、`.长度`、`.字符数` → `"整"`；`.包含?`、`.成功` → `"逻"`

**用途**：代码生成阶段的 `编译内建中缀` 会读取 `节点.中缀左.推导类型 == "整"` 来决定是否走原生指令路径。

### 5.6 代码生成 —— compileExpression

位于 [编译.xt L1471](../xuantie_compiler/编译.xt#L1471)。返回 `[regIR, type]` 二元组。

**关键路径**：

**标识符** → 查 `符号表` 找到 alloca 地址 → `load i64` → 查 `变量类型表` 获取类型 → 标量类型跳过 retain：

```xuantie
若 非 此.是标量类型(类型) {
    此.写出("  call void @xt_retain(i64 " & 寄存器 & ")")
}
返 [寄存器, 类型]
```

**中缀表达式** → 先试运算符重载（`编译中缀` L2974），无重载则走 `编译内建中缀`（L3045）：

`编译内建中缀` 有**双重路径**：

1. **静态优化路径**：`节点.中缀左.推导类型 == "整"` 且 `节点.中缀右.推导类型 == "整"` → 原生 LLVM 指令（`add`/`sub`/`mul`/`icmp`...）
2. **通用路径**：`xt_eq`、`xt_compare` 运行时函数调用

**调用表达式** → `编译调用表达式`（L2895+）：三路径分发——① 类名静态调用（`按钮.渲染(此)`）；② `函数表` 命中后直接 `call` + 脱壳；③ 动态调用（函数指针 `inttoptr`）。`printf` 有特殊的 `xt_ffi_printf` 包装。

### 5.7 关键辅助函数

| 函数         | 作用                                                     |
| ---------- | ------------------------------------------------------ |
| `符(值)`     | 将任意值安全转为 IR 字符串，遇到 `空` 返回 `"%NULL_SYMBOL_ERR"` 防止污染 IR |
| `标(值)`     | 同 `符`，用于标签                                             |
| `确保I64`    | 将任意类型值转为标记 i64（raw\_i64→装箱、i1→select、函数名→ptrtoint）     |
| `确保RawI64` | 将标记 i64 脱壳为原始 i64（`ashr i64 %x, 1`）                    |
| `是标量类型`    | 判断 `raw_i64`/`i1`/`double`                             |
| `写_释放_xt`  | 按类型释放：标量→无操作，i64→`xt_release`                          |
| `写_存储_xt`  | 按类型存储到 alloca 地址：标量存原值，i64 存标记值                        |
| `生成布尔条件`   | 将任意类型转为 i1 条件值                                         |
| `最后是返回`    | 检查输出列表最后一行是否是 `ret`/`br`/`unreachable`                 |

### 5.8 IR 输出与主函数组装

[编译程序](../xuantie_compiler/编译.xt#L416) 的末尾按顺序输出：

1. Target triple 和 data layout
2. 类型声明（`%XTObject`、`%XTString`、`%XTArray`……）
3. 外部运行时函数声明（`declare ... @xt_...`）
4. 外部 FFI 函数声明
5. 字符串常量（`@str.N = ... constant ...`）
6. 函数定义（`此.函数输出`）
7. `define i32 @main` 入口

## 7. 已知陷阱与隐形契约

### 7.1 【已修复 v0.17.5】compileFunctionDefinition 未读取参数类型注解

位置：[编译.xt L1197](../xuantie_compiler/编译.xt#L1197)

**修复前**：全部参数硬编码 `"i64"`，无视类型注解。
**修复后**：新增 `映射类型注解(注解)` 函数（`整`→`raw_i64`、`小数`→`double`、`逻`→`i1`），`编译函数定义` 的参数循环读取 `arg节点2.推导类型`，标量参数先 `ashr` 脱壳再存储，跳过 `xt_retain` 和 `追踪对象`。同时注册返回类型到 `函数返回类型表`。

### 7.2 【已修复 v0.17.5】编译赋值未处理标量变量存储

位置：[编译.xt L3232](../xuantie_compiler/编译.xt#L3232)

**修复前**：`store i64 xt_val` 存标记值。
**修复后**：标量变量存储原始值（`翻译右[0]`），非标量存标记值并 retain。

### 7.3 【已修复 v0.17.5】推导程序类型结果未被充分利用

**v0.17.5 改动**：`编译中缀` 新增标量快速路径——双方都是标量时跳过 `xt_dict_get` + `xt_string_new` 动态分发；`编译调用表达式` 的 `函数表` 分支和标识符分支新增脱壳逻辑（查 `节点.推导类型 == "整"` 或 `函数返回类型表`）。

### 7.4 形参类型注解存储在标识符节点的推导类型字段

语法分析器 [L502](../xuantie_compiler/语法分析.xt#L502) 中：

```xuantie
p节点.推导类型 = 此.解析类型()
```

这与种子编译器 `Parameter.DataType` 专用字段不同。自举编译器**复用**了 `推导类型` 字段来存原始类型名。这意味着第三趟 `推导程序类型` pass **可能会覆盖**参数上已经存好的类型注解——如果该标识符节点同时被类型推导遍历到的话。但目前没有观察到覆盖，因为推导 pass 不会遍历函数参数节点（只遍历语句和表达式）。

### 7.5 变量类型表的生命周期

`变量类型表` 是全局共享的，但在 `编译函数定义` 中会被**临时清空**并只恢复全局变量：

```xuantie
此.变量类型表 = {}
遍历 键 于 之前的符号表 {
    若 存址.截取(0, 1) == "@" { // 全局符号
        此.变量类型表[键] = 之前的类型表[键]
    }
}
```

这意味着**局部变量类型不会跨函数传播**——这是正确的行为（局部变量本来就不该跨函数可见）。v0.17.5 已通过 `函数返回类型表` 解决了跨函数返回值类型传播的需求，无需修改此机制。

### 7.6 【已修复 v0.17.5】种子编译器 `遍历` 缺少 `continueLabels` / `breakLabels`

**[compileForStatement](../compiler/llvm.go#L2528)** 在修复前缺少 `continueLabels`、`breakLabels` 和 `loopDepths` 的 push/pop。当 `续`/`断` 出现在 `遍历` 循环中时，`continueLabels` 栈顶是外层循环的目标标签，导致跳转目标错误。

**修复**：在 `遍历` 的条件分支后提前生成 `stepLabel`，同步 push 三个栈——`breakLabels(endLabel)`、`continueLabels(stepLabel)`、`loopDepths`，循环结束后 pop。

**影响**：这是 `出作用域` 两阶段 Arena 释放导致无限循环的**真正根因**——`续` 在嵌套 `当`→`遍历` 中错误跳转到外层 `当` 的条件检查，跳过 `游标 = 游标 - 1`。自举编译器的 `生成循环逻辑`（[编译.xt L937-L938](../xuantie_compiler/编译.xt#L937)）本身已有正确的 `续标签栈`/`断标签栈` 推送，不受此问题影响。

**对应测试**：[16\_控制流测试.xt](../../Test/16_控制流测试.xt) 覆盖 5 个场景：单层 `续`/`断`、嵌套 `续`/`断` 标签隔离、`当` 内嵌 `遍历` 内的 `断`。

### 7.7 【已修复 v0.17.5】符() 和 标() 的空值静默容错

这两个包装函数（L188-L212）用于防止编译器内部状态异常导致的 IR 污染。

**修复前**：遇到 `空` 值或字符串 `"空"` 时静默返回 `"%NULL_SYMBOL_ERR"` / `"NULL_LABEL_ERR"` 等错误哨兵串，这些串被写入 IR 后 clang 报错信息难以追溯原始根因。

**修复后**：遇到空值时立即 `示()` 中文错误信息 + `终 1` 强制退出，在编译器层面就暴露内部状态异常，而非让 clang 去解析非法 IR。

### 7.8 编译中缀的静态优化路径依赖推导类型

[编译内建中缀 L3050](../xuantie_compiler/编译.xt#L3050)：

```xuantie
若 节点.中缀左.推导类型 == "整" 且 节点.中缀右.推导类型 == "整" {
```

这里比较的是第三趟推导 pass 打上的 `推导类型`（值为 `"整"`），而不是实际的 IR 类型（`"raw_i64"` vs `"i64"`）。如果某个整数变量因为类型注解缺失导致 `推导类型` 是 `"未知"`，即使运行时它确实是整数，这个静态优化路径也不会被触发。

### 7.9 UAF 崩溃（玄铁.xt 主函数）

来自 INVESTIGATION\_xt.md：`玄铁.xt` 的 `主()` 函数在手动 `xt_arena_destroy(池)` 后，函数末尾编译器自动插入的 ARC 释放代码仍尝试 `xt_release(池)`，导致 Use-After-Free。

### 7.10 【已修复 v0.17.5】解析器未拦截关键字作为标识符

位置：[语法分析.xt L84-L105](../xuantie_compiler/语法分析.xt#L84)（新加的 `是保留关键字` 辅助函数）、L278（变量声明分支）、L349（遍历循环分支）

**修复前**：词法器将 `结果`、`任务`、`道`、`字节`、`整`、`字`、`逻` 等 15 个常用字映射为特殊 token 类型（非 `"标识符"`），但解析器的变量声明（`设 结果 = ...`）、遍历循环（`遍历 任务 于 ...`）、函数定义等位置只接受 `"标识符"` 类型，导致含关键字的测试报出令人困惑的"期望标识符"错误。

**修复后**：新增 `是保留关键字(类型)` 函数（检测15个关键字类型），在变量声明和遍历循环的标识符检查分支中优先拦截并输出中文友好报错：`'结果' 是玄铁语言的保留关键字，不能用作变量名或标识符。请换一个名字。`，与种子编译器的错误提示对齐。

### 7.11 【经验教训】种子编译器的 `当`+`遍历`+`若`/`续` 嵌套 IR 生成缺陷

最初在 v0.17.5 两阶段 Arena 释放中暴露，后来发现根因是种子编译器的 `compileForStatement` 缺少 `continueLabels`/`breakLabels` 栈推送（参见 7.6），已在 llvm.go L2528 修复。该缺陷不会再造成新的影响。

### 7.12 【已修复 v0.17.7】自举编译器 LLVM 类型结构体缺少 magic 字段

位置：[编译.xt L473-L478](../xuantie_compiler/编译.xt#L473)。

**修复前**：自举编译器的全部 6 种 LLVM 类型定义（XTObject/XTTring/XTTray/XTCDict/XTTFunction/XTResult）只有 2 个头部字段（`{ i32, i32 }` = ref\_count + type\_id），缺少 `magic` 字段。Go 种子编译器正确定义为 3 个头部字段（`{ i32, i32, i32 }` = magic + ref\_count + type\_id）。C 运行时 `XTObject` 结构体也有 `magic` 字段。这导致所有通过 `getelementptr` 访问 `type_id` 的代码（索引 `i32 1`）实际读到了 `ref_count`（offset 4 而非 offset 8）。

**影响链**：ref\_count 值通常是 1 → 不等于 type\_id 的任何有效值（数组=5、字典=6、字符串=3）→ 所有类型分发判断全部失败 → `遍历` 循环体从不执行、运算符重载检查永远走到 fallback 路径。

**修复（v0.17.7）**：

1. 6 种类型定义全部添加 `magic` 字段（+1 个 `i32`）
2. 28 处 `getelementptr` 字段索引全面修正：
   - XTObject type\_id：`i32 1`→`i32 2`（8 处）
   - XTTring data：`i32 2`→`i32 3`（1 处）
   - XTTring length：`i32 3`→`i32 4`（3 处）
   - XTArray length：`i32 3`→`i32 4`（3 处）
   - XTFunction funcPtr：`i32 2`→`i32 3`（1 处）
   - XTResult isSuccess：`i32 2`→`i32 3`（5 处）
   - XTResult value：`i32 4`→`i32 5`（4 处）
   - XTResult error：`i32 5`→`i32 6`（3 处）

**验证**：修复后 Test 16（控制流测试）全部通过（偶数和=6、阶段和=72），`遍历` 循环正常执行。

### 7.13 【已修复 v0.17.7】变量类型表中 raw\_i64 与标记值不一致

位置：[编译.xt L1389-L1395](../xuantie_compiler/编译.xt#L1389)（`编译变量声明语句`）、[编译.xt L3296-L3301](../xuantie_compiler/编译.xt#L3296)（`编译赋值`）。

**修复前**：`编译变量声明语句` 先将表达式值通过 `确保I64` 标签化（raw\_i64 → tagged pointer），然后存入 `变量类型表` 的类型仍是 `翻译结果[1]`（如 `"raw_i64"`）。后续读取时 `确保RawI64("raw_i64")` 做 pass-through（不脱壳），导致标签值参与原生运算、再次被标签后变成三倍编码。

**修复**：当表达式返回类型为 `"raw_i64"` 时，`变量类型表` 存储 `"i64"`（标记值类型），因为值已被 `确保I64` 转化为标记指针。`编译赋值` 同理。

**注意**：修复中 `设 类型名` 在 `若`/`否` 块内声明。玄铁语言的 `设` 是函数级作用域（评估器 `evalBlock` 不创建新 Environment），块内声明的变量在块外可见——测试 16i 通过（局部变量 x+y=7）证实了这一点。

### 7.14 【经验教训】自举编译器类型定义必须与 C 运行时严格对齐

本缺陷（7.12）揭示了自举编译器与种子编译器之间的"静默分歧"：当 Go 种子编译器正确生成 IR 时（因为它的类型定义与运行时对齐），被测产物运行正常；但当自举编译器生成的产物运行时（用自举编译器自己的类型定义），字段偏移全部错位。这种 bug 极难发现——用种子编译器编译的测试全部通过，只有用自举编译器编译的产物才暴露问题。

## 8. 代码生成 IR 示例 (v0.17.5 修复后实测)

以 `函 斐波那契(n: 整): 整 { 若 n <= 1 { 返 n } 返 斐波那契(n-1) + 斐波那契(n-2) }` 为例，自举编译器 **v0.17.5 实测** 生成的 IR：

```llvm
define i64 @"斐波那契"(i64 %"n_arg") {
entry:
  %"n" = alloca i64
  %t1 = ashr i64 %"n_arg", 1          ; 参数脱壳！
  store i64 %t1, i64* %"n"            ; 存 raw_i64
  ; 无 xt_retain！

  %t2 = load i64, i64* %"n"
  %t3 = icmp sle i64 %t2, 1           ; 原生比较！

  ; if.then: shl+or 装箱返回
  %t4 = load i64, i64* %"n"
  %t5 = shl i64 %t4, 1
  %t6 = or i64 %t5, 1
  ret i64 %t6

  ; n-1: 原生减法
  %t8 = sub i64 %t7, 1                ; 原生 sub！

  ; 递归调用 + 脱壳
  %t11 = call i64 @"斐波那契"(i64 %t10)
  call void @xt_release(i64 %t10)      ; 释放临时装箱值
  %t30 = ashr i64 %t11, 1             ; 递归返回自动脱壳！

  ; n-2: 同样原生
  %t31 = ashr i64 %t16, 1             ; 第二个递归返回值脱壳
  %t32 = add i64 %t30, %t31           ; 原生 add！
```

**v0.17.5 前（对比）：**

```llvm
; 旧 IR：参数被当堆对象 retain，比较走 xt_compare，
; 算术走 xt_sub/xt_add 运行时函数，递归返回值当 i64 传递不脱壳
; fib(30) ~365ms
```

**v0.17.5 实测**: fib(30) = **\~6ms**，fib(40) = **\~740ms**。

### 10.1 ~~Test 02 字符串测试 — 崩溃（`字节` 方法）~~ — 已修复 (v0.17.8)

**修复**：移除了 `字节()` 处理器中 `resRaw = ashr i64 resI64, 1` 这一行，直接将 `xt_string_get_byte` 返回值存入结果地址。

### 10.2 ~~Test 03 集合测试 — 崩溃（字典遍历）~~ — 已修复 (v0.17.8)

**位置**：[llvm.go L1753-L1765](../compiler/llvm.go#L1753)

**现象**：数组遍历正常完成，但到字典遍历时崩溃（ACCESS_VIOLATION）。

**根因**：Go 种子编译器没有将 `键()` 作为内置方法处理。当执行 `映射.键()` 时，走的是动态字典查找路径——尝试从字典中查找 key="键"，找不到返回 null。随后 for 循环对 null 解引用，调用 `load i32, i32* inttoptr (i64 0 to i32*)` → ACCESS_VIOLATION。

**回归验证**：Test 03 全部通过，字典遍历正确输出 `版=0.17, 地=中国, 名=玄铁`。

### 10.3 自举编译器无法编译自身（`编译.xt`）

**位置**：`.\玄铁.exe tie xuantie_compiler\xt.xt` → 编译 `编译.xt` 时报错

**现象**：Go 种子编译器编译的自举编译器（`玄铁.exe`）在编译 `编译.xt` 时，在第 3531 行附近报解析错误：`期望当前标记为 箭头，但得到 竖线`。

**影响**：当前自举链路不完整——新功能必须在 `编译.xt` 和 Go 种子编译器两处同步修改。`语法分析.xt` 需要补全匹配语法支持才能实现完全自举。

**阻塞依赖**：`语法分析.xt` 的匹配语法增强

### 10.4 种子编译器 XTString 字段偏移可疑 bug

- 成员属性弱引用（`子.父节点 = 父`）走`xt_dict_set`路径，不自动调用`xt_weak_init`
- 弱引用重赋值时旧槽位未从旁路链表中注销
- 应用层需自行`空`检查弱引用值

**回归验证**：Test 17（弱引用）全部通过退出码0：声明、重赋值、空弱引用、归零验证（覆盖强引用后弱引用自动置空）。

**ARC 行为细节**：

弱引用声明时的 ARC 操作：
1. `注册局部变量`：创建 alloca，存储值（**不 retain**——弱引用不持有所有权）
2. `xt_weak_init(slot, obj)`：将 alloca 地址注册到对象的旁路链表中
3. `xt_release(obj)`：释放表达式 load 时 `编译表达式` 自动生成的 `xt_retain`（表达式加载非标量值时会 retain→ref+1，此处 release 抵消之），对象 refcount 回到只有强引用持有的状态
4. 从作用域追踪移除 alloca：防止出作用域时编译器对弱引用变量自动调用 `xt_release`

赋值给弱引用变量时（`编译赋值`）：
- 检查 `弱引用变量集`，若命中→直接 `store`，跳过旧值 release 和新值 retain
- 注：当前未调用 `xt_weak_init` 更新槽位，重赋值后旧对象释放时不会清零新值

对象释放→弱引用归零的完整链路：
```
强引用 = 新对象（赋值）
  → xt_release(旧对象), refcount: 1→0
    → xt_free_obj(旧对象)
      → xt_weak_clear(旧对象)
        → 遍历 g_weak_slots 链表
        → *slot_addr = XT_NULL（归零！）
```

### 10.21 【已实现 v0.18.0】泛型参数可空修饰符 `?`

**位置**：[语法分析.xt L326-L338](../xuantie_compiler/语法分析.xt#L326)

**语法规则**：`?` 紧跟在一个类型后面，修饰的是它**紧邻的那个类型**。

```
节点?              ← 节点 可空
数组<节点>?        ← 数组 可空（整个数组可能为空）
数组<节点?>        ← 数组里的元素可空（数组本身存在）
数组<节点?>?       ← 数组可空，且元素也可空
字典<字, 节点?>    ← 字典里 value 可空
字典<字, 节点>?    ← 整个字典可空
```

**实现**：在 `解析泛型参数列表` 的首参数和逗号分隔参数两处，读取类型关键字/标识符后检查下一个标记是否为 `"问号"`，若是则将 `?` 拼接到参数名字符串中并消费标记。

**文档**：[03_泛型编程指南.md](./03_泛型编程指南.md) 第 3.1 节详细说明。

## 11. 内存管理架构（2026-05-21 审计）

### 11.1 整体架构

玄铁自举编译器采用 **Arena（区域分配器）** 作为核心内存管理策略。不依赖引用计数（ARC）做实际内存回收——ARC 的 retain/release 在 Arena 分配的对象上均为空操作。

```
玄铁.xt → xt_arena_new(1GB) → xt_arena_use(池) → 全局 Arena 激活
                                                    ↓
                              所有 xt_malloc / xt_string_new / xt_array_new 等
                              全部从 Arena 分配
                                                    ↓
                              编译器运行完毕 → xt_arena_destroy(池) → 一次性回收
```

### 11.2 关键参与者

| 组件 | 文件/位置 | 职责 |
|---|---|---|
| Arena 初始化 | 玄铁.xt L180-182 | `xt_arena_new(1073741824)` + `xt_arena_use(池)` |
| 对象分配入口 | runtime/xt_runtime.c `xt_malloc()` | 检测 g_current_arena → Arena 分配或 malloc fallback |
| Arena 分配 | runtime/xt_runtime.c `xt_arena_alloc()` | 设置 ref_count = XT_REF_COUNT_IMMORTAL (0x7FFFFFFF) |
| 不可销毁标记 | runtime/xt_runtime.c `xt_retain_forever()` | 将任意对象的 ref_count 强设为 IMMORTAL |
| 运行时对象 | xt_array_new/xt_dict_new/xt_string_new | 均通过 xt_malloc → Arena 分配 |
| 编译器内部容器 | 编译.xt 造() | `结果输出列表`、`函数输出列表`、`符号表`、`字符串映射` 等 20+ 字典/数组全部 Arena 分配 |

### 11.3 ARC 的实际角色

在 Arena 激活期间：
- **xt_retain**：检查 `ref_count >= XT_REF_COUNT_IMMORTAL` → 直接返回，不增加计数
- **xt_release**：同上检查 → 直接返回，不减少计数、不释放对象
- **xt_free_obj**：永远不会被调用

因此 ARC 的 retain/release 对编译器来说**纯为运行时开销，不产生任何内存管理效果**。内存管理完全依赖 Arena 的一次性回收。

### 11.4 编译器数据结构的内存生命周期

编译器 `造()` 创建的所有容器（L50-L84）：

```
此.结果输出列表 = []       // 数组
此.字符串映射 = {}         // 字典
此.符号表 = {}             // 字典
此.类型表 = {}             // 字典
此.程序缓存 = {}           // 字典
此.入口alloca列表 = []     // 数组
...（约 20 个容器）
```

全部通过 `xt_array_new` / `xt_dict_new` → `xt_malloc` → `xt_arena_alloc` 分配，引用计数为 IMMORTAL。编译器运行期间永不释放，随 Arena 销毁一次性回收。

### 11.5 预扫描与程序缓存的内存行为

预扫描引入（`预扫描引入`）处理 4 个被引入模块（词法.xt、抽象语法树.xt、语法分析.xt、编译.xt）。当前使用 **程序缓存** 策略：解析一次、缓存 AST、代码生成时复用。

AST 全部在 Arena 中。类型表通过指针引用 AST 中的类型定义节点（不复制）。4 个模块 + 1 个主文件的完整 AST 均在 1GB Arena 内。

**注意**：bc78c6a commit 曾尝试将 `程序缓存` 改为 `源码缓存`（只缓存源文本、不缓存 AST）以减少预扫描内存，但这导致代码生成阶段重新解析、语义分析添加的 `此` 参数丢失。**已回退到程序缓存**。

### 11.6 隐式契约与设计不变量

| 契约 | 说明 | 违反后果 |
|---|---|---|
| **Arena 必须在使用前激活** | `xt_arena_use(池)` 必须在 `造 编译器()` 之前调用 | 对象落入 malloc，ARC 实际生效，可能 double-free |
| **`此` 参数由语义分析添加** | `解析型成员` 遍历类型表，为每个方法前插 `此` 参数 | 代码生成时 `此` 找不到 → 编译错误 |
| **程序缓存 AST 与类型表共享节点** | 类型表条目是程序缓存 AST 的子节点引用，不是副本 | 如重新解析，类型表指针悬挂 |
| **入口alloca列表 必须 save/restore** | `编译函数定义` 递归调用时需保存/恢复外层列表 | 外层函数的 alloca 被内层覆盖 → LLVM 支配性错误 |
| **`返 [expr, type]` 不可内联数组字面量** | 产物中该模式返回值为空；必须先 `设` 再 `返` | 调用者收到 XT_NULL → 编译失败 |
| **禁用运算符重载于 `==`/`!=` 非标量** | 运算符重载产生内联 alloca i64，在 3+ `若/抑` 条件下堆损坏 | 运行时 xt_check_obj 失败 |
| **字符串索引必须处理 type=3** | `s[i]` 需检查 XT_TYPE_STRING(3)，调用 `xt_string_get_char` | 误调 `xt_array_get` → 访问违例 |
| **所有 alloca 需配 `store i64 0` 零初始化** | entry alloca 列表和参数 alloca 后跟随 store i64 0 | 未初始化内存被 ARC release 时读到垃圾值 |
| **`解析中缀` default case 使用 `优先级-1`** | `==`/`!=` 在优先级=4 时递归返回崩溃 | 解析器访问违例 |

#### 12.2.2 `==`/`!=` 运算符重载导致 3+ `若/抑` 条件下堆损坏

**根因**：`编译中缀表达式` 对 `==`/`!=` 走运算符重载路径（`xt_dict_get(左, "_等_")`），该路径创建内联 alloca i64（`resAddr`），在 3+ `若/抑` 条件评估时产生堆损坏。

**修复**：在运算符重载入口处添加 early-return：当 `节点.中缀运算符 == "==" 或 "!="` 时，直接调用 `编译内建中缀`（其 `==` 路径天然走 `@xt_eq`，不创建内联 alloca）。

**置信度**：高（90%）——修复后 3 条件测试从崩溃变为正常。

#### 12.2.3 `返 [expr, "type"]` 产物返回空值

**根因**：`编译返回语句` 调用 `编译表达式(节点.返回值)` 递归编译。产物中将 `返 [xt_val, "i64"]` 的数组字面量传递给 `返` 时，递归 `编译表达式` 在产物中返回空值。分离为独立变量后正常。

推断：产物中内联数组字面量在 `匹配` 语句的 case 体内创建时，寄存器/变量作用域存在问题。

**修复**：全局替换 `编译.xt` 中 74 处 `返 [变量, "type"]` 为 `设 _r = [变量, "type"] ; 返 _r`。

**置信度**：中（65%）——修复有效但根因未完全理解。可能与产物中 `匹配` case 体的寄存器分配有关。

#### 12.2.4 字符串索引 `s[i]` 不处理 XT_TYPE_STRING

**根因**：索引编译代码（编译.xt L1983-2004）仅检查 type_id==6（字典），否则调用 `xt_array_get`。对 type_id==3（字符串）误调 `xt_array_get`，将 XTString 强制当作 XTArray 解析，导致访问违例。

**修复**：在 dict 和 array 之间插入 string 检查分支，调用 `xt_string_get_char`。

**置信度**：高（95%）。

#### 12.2.5 `解析中缀` default case 在优先级=4 递归崩溃

**根因**：`解析表达式(优先级)` 在 infix 循环中递归调用自身。`==`/`!=` 的优先级为 4，与 `{`（优先级 0）的 break 判断 `4 >= 0` 正确，但递归返回时崩溃（仅在自举产物中）。

**修复**：将 `解析表达式(优先级)` 改为 `解析表达式(优先级 - 1)`。

**置信度**：中（60%）——修复有效但精确根因未完全定位。可能与 `4 >= 0` 的比较在产物中生成的代码有关。

#### 12.2.6 `入口alloca列表` 递归覆盖

**根因**：`编译函数定义` 递归调用时重置 `此.入口alloca列表 = []`，外层函数（如 `编译程序`）积累的入口 alloca 被覆盖。

**修复**：在 `编译函数定义` 开头保存 `之前的入口alloca列表`，结尾恢复。

**置信度**：高（85%）。

#### 12.2.7 繁体字 `設` 导致 Go seed 链接失败

**根因**：修复脚本错误使用了繁体中文 `設` 代替简体 `设`，导致 Go seed 生成的 IR 引用不存在的 `@設` 符号。

**修复**：`設` → `设`。

#### 12.2.8 `解析遍历循环` 缺少 `设` 声明

**根因**：`语法分析.xt` L539 的 `变量名 = ...` 声明了 `变量名` 但漏写 `设`。Go seed 宽容处理，自举编译器要求显式声明。

**修复**：`变量名 = ...` → `设 变量名 = ...`。

### 14.2 已知取舍（可能给未来埋坑）

| 来源 | 取舍 | 风险 | 建议 |
|------|------|------|------|
| C4 | Arena 活跃时对原本 malloc 的数组 elements 扩容，旧缓冲区**泄漏** | arena 模式在程序启动后通常不变，实践中极少触发 | 后续可添加 fallback free 路径 |
| C4 | `XTArray.elements_in_arena` 加在 struct 末尾 | 自举编译器 IR 中 `%XTArray` 类型定义为 6 字段，C 侧现为 7 字段。若未来编译器直接用 GEP 访问数组字段可能偏移错误 | 目前编译器通过 C 运行时函数访问数组，不直接 GEP，安全 |
| C7 | JSON 序列化转义缓冲区按最坏情况分配（`length*6+3`），大字符串内存消耗高 | 10MB 字符串需要 60MB 临时缓冲区 | 后续可改用两遍扫描（第一遍计大小，第二遍写入） |
| C2 | `xt_json_deserialize` 对未闭合字符串"原样返回"而非报错 | 调用者可能误用损坏数据 | 后续应返回 `XT_NULL` 或 Result 错误 |
| H21-H22 | 弱引用 release 后依赖 `xt_free_obj→xt_weak_clear` 清槽——**单线程下安全，多线程下 `xt_weak_clear` 遍历全局链表无锁** | 当前玄铁为单线程，多线程后需加锁 | 后续引入线程时优先修此 |
