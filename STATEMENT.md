# 立场声明

我们做玄铁工具链,不是因为上游玄铁不好,而是因为一门语言要真正可用,**工具链体验和语言本身一样重要**。

## 我们要什么

- **开源、免费、跨平台** — 不收费、不绑定单一平台、不搞闭源壳子
- **git 友好** — 项目就是普通目录 + `xuantie.toml`,依赖用 git 拉取和锁定,不依赖中心化包注册表也能工作
- **cargo/uv 风格** — `new` / `run` / `build` / `test` / `add` / `check` / `lint` / `clean`,熟悉的味道,母语的关键字
- **可复现构建** — `xuantie.lock` 锁定依赖 commit,团队/CI 上构建一致

## 我们不做什么

- 不重新发明玄铁语言 — 语言实现完整继承自上游,我们只做包裹层
- 不做 IDE/编辑器 — 那是另一件事,VSCode 插件等独立解决
- 不做包注册表 — 依赖直接走 git + tag/branch/rev,够用且去中心化
- 不改上游编译器核心逻辑 — 除非是为跨平台编译做的必要隔离 (如 Windows FFI 的 build tag 分流)

## 致上游

玄铁语言由 **问号盒** 独立设计与实现,从 2026 年 4 月起步,一个人 + AI 辅助,做到自举编译器、LLVM 后端、铁铺包管理器。这份工作量与坚持值得尊重。

本工具链 fork 自上游,完整保留了玄铁语言实现的全部源码,按 MIT 许可继续传播并声明归属。工具链 CLI 部分是我们在公有领域下贡献的新代码,目的是让玄铁更容易被安装、被使用、被集成进现代开发流。

如果上游愿意,我们乐于把工具链贡献回上游。

## 致用户

玄铁仍在 0.x 阶段,语法可能变动,不建议生产使用。但你可以现在就用 `xuantie new` 起一个项目,体验一门"骨子里就是中文"的语言是什么感觉。

```bash
cargo install --path cli
xuantie new 我的项目
cd 我的项目
xuantie run
```

输出:`你好，玄铁！`
